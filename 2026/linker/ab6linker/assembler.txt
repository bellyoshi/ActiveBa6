アセンブラ（Assembler）の自作へようこそ！コンパイラ・リンカー・アセンブラという3大ツールチェーンの中で、アセンブラは「最もシンプルかつ直感的で、作っていて楽しい」パーツです。アセンブラが行う仕事の本質は、「テキストの命令（push 0 や mov eax, ebx）を、対応するx86マシン語（バイナリ）に1対1〜1対数個で翻訳し、COFF（.obj）形式にパッキングすること」です。自作アセンブラの全体像と、具体的な作成アプローチを整理してみましょう。1. 自作アセンブラの全体の処理フローアセンブラはテキストファイルを先頭から1行ずつ読み込んで処理していきます。[ テキスト (.asm) ]
   │  "mov eax, 1" / "call _ExitProcess"
   ▼
1. 字句解析・構文解析 (Lexer / Parser)
   │  命令名 (mnemonic) と オペランド (register / immediate / memory) を分類
   ▼
2. 命令の機械語変換 (Code Generation)
   │  "mov eax, 1"  → B8 01 00 00 00
   │  "call label"  → E8 [未定のアドレス] + リロケーション登録
   ▼
3. シンボルテーブルの構築
   │  "_main:" などのラベルがコード内の何バイト目にあるかを記録
   ▼
4. COFF (.obj) フォーマットへの出力
   │  マシン語バイナリ＋COFFヘッダー＋シンボル表＋リロケーション表を書き出す
   ▼
[ オブジェクトファイル (.obj) ]
2. アセンブラ作りの心臓部：「x86マシン語の組み立て」x86（32bit）命令のマシン語変換は、命令のパターンによって難易度が変わります。① 最も簡単：オペランドなし命令単一の1バイトに変換するだけです。nop  $\rightarrow$ 90ret  $\rightarrow$ C3② 次に簡単：レジスタへの即値代入（32bit）mov レジスタ, 数値 のパターンです。x86ではレジスタごとに専用の1バイト命令（Opcode）が用意されています。mov eax, 0x12345678 $\rightarrow$ B8 78 56 34 12 (B8 + 32bit即値)mov ecx, 0x12345678 $\rightarrow$ B9 78 56 34 12 (B9 + 32bit即値)③ アセンブラ最大の難所：ModR/M バイトの計算mov [ebp-4], eax や add eax, ebx のように、「レジスタ同士」や「メモリとレジスタ」のやり取りをする命令です。x86では、オペランドの組み合わせを ModR/M（モッド・アールエム） という1バイトのビットフィールドで表現します。ModR/M バイトの構造 (8 bits)
[ Mod (2 bits) ] [ Reg/Opcode (3 bits) ] [ R/M (3 bits) ]
Mod (モード): メモリ参照の形式（レジスタ直接、[ebp+offset8]、[ebp+offset32] など）Reg: 転送元（または転送先）のレジスタ番号 (eax=0, ecx=1, edx=2 ...)R/M: もう一方のレジスタ番号またはメモリアドレス指定💡 コツ：最初は「使う命令のパターン」だけに絞る！x86の全命令に対応しようとするとIntelのマニュアル（数千ページ）に潰されます。最初はコンパイラが出力する 10種類ほどの基本命令（push, pop, mov, add, sub, cmp, jmp, call, ret） だけをターゲットに限定するのが成功の秘密です。3. アセンブラ実装の2パス（2-Pass）アルゴリズムアセンブラを作るときに必ず直面するのが「前方参照（まだ登場していないラベルへのジャンプ）」問題です。コード スニペット    jmp _future_label   ; この時点では _future_label が何バイト目にあるか分からない！
    nop
_future_label:
    ret
これを解決するために、アセンブラはコードを2回読み込み（2-Pass）します。1.1st Pass (位置計算):アドレス計算とラベル登録。テキストを1行ずつパースし、「各命令が何バイトの機械語になるか」だけを計算してロケーションカウンタ（位置ポインタ）を進める。_label: を発見したら、その時点の「アドレス（バイト数）」をシンボルテーブルに保存する。2.2nd Pass (機械語出力):バイナリ生成とリロケーション。もう一度テキストを先頭から読み込み、今度は実際の機械語バイナリを出力する。jmp _future_label が出てきたら、1st Pass で作成したシンボルテーブルから目的のアドレスを引き出して相対差分を計算・埋め込む。4. 自作アセンブラ開発のロードマップ自作リンカーの時と同様に、段階を踏んで作ると確実に動くものが完成します。ステップ1：超最小アセンブラ（固定命令のみ）入力：push 0 / call _ExitProcess だけが書かれたテキスト処理：push 0 を見つけたら 6A 00 をバッファに追加。call _ExitProcess を見つけたら E8 00 00 00 00 を追加し、COFFリロケーションテーブルに「_ExitProcess への参照」を1件登録。COFF構造体（前回作成した知識がそのまま使えます！）に流し込んで .obj を書き出す。ステップ2：数値パラメータとレジスタの解析push 42 $\rightarrow$ 6A 2Amov eax, 1 $\rightarrow$ B8 01 00 00 00文字列からレジスタ名（eax, ebx 等）や10進数・16進数をパースする処理を追加。ステップ3：ラベルとジャンプ（1st Pass / 2nd Pass）_label: 構文の解析と、jmp _label / je _label などの相対ジャンプ命令の補正計算を実装。次の一歩アセンブラが完成すれば、自分で吐き出した .obj を「前に作った自作リンカー」に食わせて、一気通貫で .exe を生み出すことができるようになります！まずは「push 0 と call だけを変換して .obj を吐き出す最小のプログラム」の設計から始めてみますか？
